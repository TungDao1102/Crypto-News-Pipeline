Purpose: Central health aggregation point consumed by the `/health` command and alert delivery system. Each module registers a health check callback. The collector runs all checks concurrently with individual timeouts. Alert cooldown prevents duplicate alert storms during cascading failures.

    Create `src/health.py` with:

    1. Import `asyncio`, `logging`, `time`, `datetime`, `Callable`, `Awaitable`, `Any`, `dataclass`.
    2. Module-level constant `ALERT_COOLDOWN_SECONDS = 1800`.
    3. Type alias `HealthCheckFn = Callable[[], Awaitable[dict[str, Any]]]`.
    4. `CheckEntry` dataclass with fields: `check_fn: HealthCheckFn`, `timeout: float = 5.0`.
    5. `HealthCollector` class:
       - `__init__(self, alert_cooldown: float = ALERT_COOLDOWN_SECONDS)` — initialize `self._checks: dict[str, CheckEntry]`, `self._alert_cooldowns: dict[str, float]`, set `self._alert_cooldown_duration`, set `self._bot = None`, `self._admin_chat_id: str | None = None`.
       - `set_bot(self, bot, admin_chat_id: str) -> None` — store bot reference for alert delivery.
       - `register(self, name: str, check_fn: HealthCheckFn, timeout: float = 5.0) -> None` — add check with warning if name already registered.
       - `can_send_alert(self, event_type: str) -> bool` — check cooldown dict; if not present or cooldown expired, update timestamp and return True; else return False. Use `time.monotonic()`.
       - `send_alert(self, event_type: str, message: str) -> None` — check `can_send_alert`, log debug if suppressed, send via `self._bot.send_message(chat_id=self._admin_chat_id, text=f"⚠️ {message}")` with try/except logging.
       - `get_report(self) -> dict[str, Any]` — run all checks concurrently via `asyncio.gather()` with `asyncio.wait_for()` per check. Each check returns `(name, {"status": "ok"/"timeout"/"error", "data"/"error": ...})`. Overall status is "healthy" if all ok, else "degraded". Include `timestamp` (UTC ISO 8601 with Z suffix).

    Design decisions (per D-06, D-03):
    - Initialize `_alert_cooldowns` with all expected event types at startup (timestamp = time.monotonic()) to prevent startup alert storm (per RESEARCH.md Pitfall 2)
    - The five event types: `"source_disconnect"`, `"all_models_exhausted"`, `"queue_overflow"`, `"binance_daily_limit"`, `"publisher_permanent_error"`
    - Alert cooldown is first-only-per-event-type, flatly suppressing all duplicates within the window (per D-03, RESEARCH.md "not TokenBucket")
    - Health checks are always async — document that callbacks must be `async def`

    Write `tests/test_health.py`:
    - `test_register_and_report` — register two mock checks, run get_report(), verify both results present, overall is "healthy"
    - `test_check_timeout` — register a check that sleeps 10s with 0.01s timeout, verify it returns with status "timeout"
    - `test_check_error` — register a check that raises Exception, verify status is "error"
    - `test_alert_cooldown_suppresses_duplicates` — send same event_type twice within cooldown, verify only first returns True
    - `test_alert_cooldown_allows_after_window` — mock time to advance past cooldown, verify second alert returns True
    - `test_startup_alert_storm_prevention` — verify all event types initialized in _alert_cooldowns at create time, so can_send_alert returns False initially

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | HealthCollector.get_report() | mitigate | Report only contains registered module status data; no secrets, no file paths, no IPs. The consumer (/health command) must strip any leaked data |
| T-05-02 | Denial of Service | Alert cooldown | mitigate | Cooldown uses time.monotonic() — immune to system clock changes. Cooldown resets after 30 min regardless of alert count. Multiple event types ensure at least one alert path is available |
| T-05-03 | Information Disclosure | send_alert message content | mitigate | Alert messages must never include bot tokens, API keys, or internal file paths. This plan only defines send_alert — the callers in Plan 06 are responsible for message content |
| T-05-SC | Tampering | pip install (none needed) | accept | Zero new dependencies — all stdlib. No package legitimacy gate needed |
</threat_model>
